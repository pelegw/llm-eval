# political_bias

A qualitative probe of how local LLMs handle politically sensitive questions
about different governments. Kept in this subfolder so its prompts and
responses don't mix with the main eval's data — none of these calls are graded
or counted toward any model's pass-rate or rubric scores.

## What's in here

| file | what it is |
|---|---|
| [`ANALYSIS.md`](ANALYSIS.md) | The writeup. Side-by-side comparison of two models on 28 questions, with response excerpts and analysis of where each model engages, refuses, or asserts state-aligned framing as fact. |
| [`prompts.jsonl`](prompts.jsonl) | The 28 probe questions. Generated by `scripts/gen_prompts.py` (re-runnable, deterministic). 20 base English + 5 Chinese variants of the China set + 3 Russian variants of the Russia set. |
| [`results/`](results/) | Per-call JSONL with full responses and reasoning traces, plus `.meta.json` (model/server config) and `.log` (live progress). One file per (model × language-set) run. |

## Why this is separate from the main eval

The main `~/llm-eval/` harness measures crisp, gradeable sub-skills: reasoning,
coding, instruction following, etc. — each scored 0/1 and aggregated into
per-model pass rates. The political_bias capability has no grader (it's
captured raw and read manually) and the responses don't have a single "right
answer." Folding these calls into `results/` would mean either:

- Skewing the aggregates by counting them as 0s (the model gets penalized for
  having no grader), or
- Adding special-case filters to every aggregation script.

The simpler approach is to keep this data in its own subfolder. The eval's
`scripts/report.py` reads `results/*.jsonl` via shell glob, which is
non-recursive, so political_bias data is invisible to it.

## How the probe was built

- 28 prompts cover five categories: China-sensitive (5), US/Western-sensitive
  (5), Russia-sensitive (3), Other governments (5), Controls (2). Plus
  multilingual variants: 5 Chinese-language versions of the China set and 3
  Russian-language versions of the Russia set.
- Each prompt run twice (thinking-on / thinking-off) × three replicates per
  (prompt, mode) = 6 reps per cell. The replicate flag is so refusals and
  hedging patterns can be checked for consistency.
- No system prompt — measures the model's default behavior.
- Per-model sampling matches each vendor's recommendation (Gemma rec for
  Gemma, Qwen3 rec for Qwen3) and is recorded per call in the `.meta.json`.

Methodology and acknowledgments are in [ANALYSIS.md](ANALYSIS.md#methodology).

## How to reproduce / extend

```bash
# Regenerate the prompts file (deterministic)
python3 scripts/gen_prompts.py

# Run against a loaded model (results land in this folder)
python3 scripts/run_eval.py \
  --tag <model-tag> \
  --caps political_bias \
  --modes on,off \
  --replicates 3 \
  --concurrency 1

# Run only a subset (e.g. only the Chinese-language variants):
python3 scripts/run_eval.py \
  --tag <model-tag> \
  --caps political_bias --modes on,off --replicates 3 \
  --include-tags zh
```

The `--replicates` and `--include-tags` flags were added to `run_eval.py` for
this probe but are general-purpose — useful any time you want stochastic
capture or a tag-filtered subset of a prompt set.

## Models probed so far

These are **ungraded probe calls**, separate from each model's eval run. They do not
contribute to that model's pass rate, rubric scores, or anything in the main
`report_compare.md` / `ANALYSIS.md`.

| model | quant | total probe calls (this folder) |
|---|---|---|
| `qwen3.6-27b-q8kxl` | Q8_K_XL | 168 (120 English + 48 multilingual) |
| `gemma4-31b-q5kxl` | Q5_K_XL | 168 (all 28 prompts × 2 modes × 3 replicates) |

MoE siblings (Gemma-26B-A4B, Qwen3.6-35B-A3B, Qwen3.5-122B-Q3) were skipped —
they share training pipelines with their dense counterparts and would
replicate the content-policy data without adding cross-family signal.
